Acumatica Events Explained — RowSelected, RowInserted, and Friends

A clear, example-driven walk through every Acumatica graph event you actually use — RowSelected, RowInserting, RowUpdated, FieldDefaulting, FieldUpdated, RowPersisting — with the right one for the job.

John KihiuJun 2, 20268 min read

Pick the wrong Acumatica event and you will have phantom bugs for months. Pick the right one and your code reads naturally. This guide is the field-tested mapping.

The event timing diagram

  1. FieldDefaulting — fires when the field needs a value (only if no value is present).
  2. FieldUpdating — fires when the user enters a value, before it is stored.
  3. FieldUpdated — fires after the field is stored in the cache.
  4. RowInserting / RowUpdating — fires when a row is about to be saved.
  5. RowInserted / RowUpdated — fires after the row is saved.
  6. RowPersisting — fires during the actual database commit. Last chance to touch the row.
  7. RowPersisted — fires after the database commit completes.
  8. RowSelected — fires whenever a row is loaded into the cache for UI display.
The single most common mistake Putting heavy logic in RowSelected. It fires for every row on every render. The screen becomes a slideshow. Move your logic to RowPersisting or a graph action.
