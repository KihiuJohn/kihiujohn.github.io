Acumatica Performance Tuning — The Definitive Guide

A complete reference to Acumatica performance tuning — application, database, IIS, Azure, and Acumatica-specific patterns that make a slow tenant fast.

John KihiuJun 11, 202618 min read

Acumatica is fast out of the box. It becomes slow when one of three things happens: the SQL Server is undersized, the IIS application pool is misconfigured, or a customization is doing the wrong thing. This guide is the systematic approach to find and fix all three.

1. Establish a baseline

Before you tune anything, capture a baseline. Use SQL Server's Performance Monitor counters and Acumatica's built-in performance logs. Without a baseline, you cannot prove the fix worked.

2. The four layers to inspect

  1. SQL Server — wait stats, query plans, index usage, blocking.
  2. IIS / application pool — recycling settings, thread count, queue length.
  3. Acumatica application — request handling, cache hit rate, slow request log.
  4. Custom code — slow graph events, large data contexts, expensive joins.

3. SQL Server — start here

Acumatica is a SQL Server application. The most common performance problems are SQL problems. Use:

  • sys.dm_exec_query_stats — find the most expensive queries.
  • sys.dm_exec_wait_stats — see where SQL Server is waiting.
  • Query Store (Acumatica 2022 R2+) — historical query performance over time.
  • Execution plans — the single most useful tool for query tuning.

4. Indexing

Most slow queries are missing-index queries. The rule:

  • Index the join columns.
  • Index the filter columns.
  • Index the foreign keys on every DAC.
  • Don't over-index — every index slows writes.

5. Application pool

IIS application pool misconfiguration is the second most common cause. Set:

  • Recycling: every 24 hours, or never (avoid the 5 AM recycle).
  • Idle timeout: 0 (no idle timeout).
  • Queue length: 5000 (default is too low for heavy load).
  • Worker process: one per core, no more.

6. Acumatica configuration

From the System Monitor screen, check:

  • Cache size — should be > 80% hit rate.
  • Connection pool — should not be saturated.
  • Long-running requests — investigate anything over 30 seconds.

7. Customization patterns that hurt

  • Heavy logic in RowSelected — fires for every row.
  • Loops in event handlers that fire on every save.
  • Excessive database calls where one would do.
  • Not using the cache invalidation properly.
  • Reports that fetch every row and filter in the report.

8. Azure-specific tuning

If you are on Azure VM or Azure SQL, additional considerations apply:

  • VM size — Standard D8s v3 minimum for production.
  • Premium storage for SQL data and log disks.
  • SQL Managed Instance vs Azure SQL — Managed Instance is closer to on-prem.
  • Application Insights for distributed tracing.

9. Load testing

Before declaring the tenant tuned, load test it. Tools:

  • Visual Studio Load Test (built in).
  • JMeter (open source).
  • k6 (modern, scriptable).
  • Acumatica's own performance testing tools (since 2024 R2).

10. Continuous monitoring

Tuning is not a one-time event. Set up:

  • SQL Server wait stats alerts.
  • Application pool recycling alerts.
  • Acumatica slow request log review.
  • User-reported "the system is slow" trend.

11. Common quick wins

  1. Add the missing index. 80% of slow queries are missing-index queries.
  2. Move heavy logic out of RowSelected.
  3. Disable the IIS 5 AM recycle.
  4. Increase the application pool queue length.
  5. Check the Acumatica cache hit rate.
Don't tune what isn't broken The most expensive thing in performance work is changing the wrong thing. Baseline first, measure second, change third, measure again.
